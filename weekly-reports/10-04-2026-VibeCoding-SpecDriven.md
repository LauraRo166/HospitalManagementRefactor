## 1. What Is Vibe Coding?

**Vibe Coding** is an AI-assisted development practice in which the developer describes what they want to build in natural language, and an AI model (such as Claude, GitHub Copilot, or ChatGPT) generates the corresponding code. The developer acts more as a "supervisor and validator" than as a line-by-line coder.

The term was coined by Andrej Karpathy in 2025 to describe a style of programming where developers "follow the vibe" and rely on the model for implementation.

### Main Characteristics

- Developers describe intent, not detailed technical instructions.
- Code is generated by AI and reviewed by humans.
- It enables very fast iteration from idea to prototype.
- It lowers the entry barrier to software development.

---

## 2. What Is Spec-Driven Development (SDD)?

**Spec-Driven Development** is a methodology where software development starts with explicit and formal specifications before writing code. Specs act as a contract between what the system should do and how it should be implemented.

### Main Characteristics

- Specifications are written before code (similar to TDD, but at design level).
- Specs may include API contracts (OpenAPI/Swagger), formal user stories, data schemas, or acceptance criteria.
- It improves communication across teams (frontend, backend, QA, design).
- Code is generated or implemented from the specifications.

---

## 3. Relationship Between Vibe Coding and SDD

| Aspect | Vibe Coding | Spec-Driven Development |
|---|---|---|
| Starting point | Informal description / "vibe" | Formal, structured specification |
| Developer role | Guides and validates AI output | Defines and validates the spec |
| Speed | Very high (rapid prototyping) | High, with more initial rigor |
| Risk | Technical debt if review is weak | Lower, specs reduce ambiguity |
| Tools | LLMs (Claude, Copilot, GPT) | OpenAPI, Gherkin, JSON Schema |
| Typical use | Prototypes, MVPs, isolated tasks | Critical systems, larger teams |

When combined, Vibe Coding + SDD allows teams to move fast with AI while keeping quality through clear and verifiable specifications.

## 4. Advantages

### Vibe Coding

- **Extreme speed** in code and prototype generation.
- **Reduced cognitive load** for repetitive development tasks.
- **Accessible** for developers with lower technical experience.
- **Fast iteration** to compare multiple solutions in minutes.

### Spec-Driven Development

- **Clarity and alignment** across all project stakeholders.
- **Fewer bugs** by defining behavior before implementation.
- **Testing support** since specs can become test cases directly.
- **Living documentation** that stays tied to system behavior.

---

## 5. Challenges and Limitations

### Vibe Coding

- **Invisible technical debt**: generated code may work but be poorly structured.
- **Auditability issues**: if developers do not understand generated code, maintainability drops.
- **Security risks**: AI can generate vulnerable code if not reviewed carefully.
- **Skill erosion** if teams over-rely on AI.

### Spec-Driven Development

- **Initial overhead**: writing formal specs takes time before coding starts.
- **Outdated specs** risk when project scope changes quickly.
- **Team discipline required** to keep spec and code aligned.
- **Learning curve** for tools such as OpenAPI and Gherkin.

## 6. Applied Analysis: Hospital Management System

### Vibe Coding in This Project

During the development of this Hospital Management System, several opportunities were identified where **Vibe Coding** could have accelerated work:

- **DTO and entity generation**: classes like `OpdDetails`, `Patient`, and `Doctor` follow repetitive structures that an LLM can generate from a structured prompt.
- **God Class refactoring**: AI can propose responsibility splits for controllers that accumulate too much logic.
- **Unit test generation**: from expected behavior descriptions, an LLM can draft test cases for DAO and service methods.

### Spec-Driven Development in This Project

The system currently lacks formal specifications. Adopting SDD could provide these benefits:

- **API contracts**: there are no formally documented REST endpoints; OpenAPI could drive client/server contract generation.
- **Module acceptance criteria**: Doctor, Receptionist, and Administrator modules do not have formal role-based behavioral specs.
- **Database schema as specification**: using the MySQL schema as a source of truth for Hibernate entity generation would reduce inconsistencies.

### Combined Adoption Proposal

1. Define a spec (Gherkin acceptance criteria or structured natural-language requirement)
2. Use Vibe Coding (LLM) to generate code from the spec
3. Review and validate generated code against the spec
4. Commit with automated tests that verify the spec

---

## 7. Conclusions

- **Vibe Coding** is a powerful accelerator, but it requires active supervision to prevent technical debt and security issues.
- **Spec-Driven Development** provides the structure needed to ensure generated code (human- or AI-produced) is correct, maintainable, and testable.
- The **combination of both practices** is especially valuable in this project, where improving existing code quality and delivering new features must happen in parallel.
- For this Hospital Management System, adopting SDD for module contracts and using Vibe Coding for refactoring and test generation would significantly improve development quality and speed.
